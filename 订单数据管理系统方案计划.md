# 订单数据管理系统方案计划

生成日期：2026-07-01

## 1. 建设目标

建设一个面向月度订单 Excel 数据的订单数据管理系统，用于长期存储订单 SKU 明细，并围绕产品、店铺、平台、客户、区域、物流和利润等维度分析业务情况。

本系统优先解决四类问题：

1. 每月百万级 Excel 订单数据可稳定导入、校验、去重、追溯。
2. 历史订单数据永久留存，支持按月、按年持续查询和导出。
3. 产品业务表现可量化，包括销量、销售额、成本、利润、利润率、区域分布、平台店铺表现等。
4. 系统结构保持简单，避免过早引入分库分表、复杂数仓或高成本组件。

利润建议由系统按统一公式计算：

```text
利润 = 应收合计分摊 - 成本 - 运费 - 辅料 - 分摊费用
```

Excel 中的利润字段可作为导入提示字段，用于比对系统计算结果是否一致；不一致时只提示，不阻止入库。正式入库以导入明细字段按系统公式计算利润。

## 3. 总体架构

参考现有 MySQL 落地方案，建议采用：

```text
单库 + 单业务宽表 + 按月 RANGE 分区 + 临时表批量导入 + 导入日志追溯 + 业务分析看板
```

核心表规划：

| 表名 | 用途 | 是否持久化 |
| --- | --- | --- |
| t_order_sku_detail | 订单 SKU 明细主表，按发货月份分区 | 是 |
| t_import_log | 导入批次日志、文件记录、成功失败统计 | 是 |
| tmp_order_import | 月度 Excel 临时导入、清洗、校验、去重 | 否 |

建议后续按需要增加轻量维表，例如产品维护表、客户维护表、平台店铺维护表，但第一阶段不强制拆表，避免增加导入和对账复杂度。

## 4. 核心业务功能

### 4.1 数据导入管理

功能要求：

1. 上传月度 Excel 文件。
2. 识别模板字段，校验字段是否完整、顺序是否匹配。
3. 将数据先写入临时表，不直接进入正式业务表。
4. 校验必填项：订单编号、商品链接SkuId、平台渠道、店铺、产品名称、货品编号、数量、应收合计分摊、发货时间。
5. 校验数值项：数量、收入、成本、费用不能出现不合理负数。
6. 校验日期项：发货时间不能为空，且应落在本次导入月份范围内。
7. 校验重复项：本批次内重复、与历史数据重复。
8. 生成导入报告：总行数、成功行数、失败行数、重复行数、异常原因。
9. 人工确认后批量写入正式表。
10. 写入 t_import_log，保留文件名、批次号、操作人、导入时间和备注。

### 4.2 订单明细查询

基础筛选条件：

1. 发货时间范围。
2. 平台渠道、店铺。
3. 部门、客户编号。
4. 大类、产品名称、货品编号、SKU。
5. 省份、城市。
6. 订单编号、原始单号、物流单号。

查询规范：

所有大数据量查询必须带发货时间范围，默认按月查询，避免跨多年全表扫描。

### 4.3 产品业务分析

核心指标：

| 指标 | 计算口径 |
| --- | --- |
| 销量 | sum(qty) |
| 销售额 | sum(share_receivable) |
| 总成本 | sum(cost) |
| 总费用 | sum(express_fee + logistics_fee + freight + aux_material + share_cost) |
| 利润 | sum(profit) |
| 利润率 | sum(profit) / sum(share_receivable) |
| 客单 SKU 数 | count(distinct sku_id) 或按订单聚合 |
| 订单数 | count(distinct order_no) |

分析视角：

1. 产品排行榜：按销量、销售额、利润、利润率排序。
2. 产品趋势：按月查看单品销量、收入、利润变化。
3. 大类分析：比较不同产品大类的收入、利润贡献。
4. 低利润预警：识别利润率低于阈值的产品。
5. 高成本产品：识别成本或物流费用占比异常的产品。
6. 平台产品表现：同一产品在不同平台、店铺的表现对比。
7. 区域产品表现：同一产品在不同省市的销量和利润分布。

### 4.4 店铺与平台分析

指标：

1. 各平台销售额、销量、利润、利润率。
2. 各店铺销售额、销量、利润、利润率。
3. 平台和店铺月度趋势。
4. 店铺 TOP 产品和亏损产品。
5. 平台费用结构对比。

### 4.5 客户与区域分析

指标：

1. 客户销售额、订单数、利润贡献。
2. 省份、城市销售热力分布。
3. 区域产品偏好。
4. 区域物流费用和利润率对比。

### 4.6 成本与物流分析

指标：

1. 快递费、物流费、运费、辅料、分摊费用占收入比例。
2. 不同物流方式的成本和利润表现。
3. 异常物流费用订单清单。
4. 产品维度的综合费用率。

## 6. 数据库设计要点

主表使用 t_order_sku_detail，按 ship_time 月度 RANGE 分区。这样月度查询时 MySQL 可以自动裁剪分区，长期保留数据也不会让常规月度查询持续变慢。

需要特别注意：MySQL 分区表的唯一索引必须包含分区字段。因此如果使用按 ship_time 分区，单独建立 `unique(order_no, sku_id)` 在 MySQL 中不可行。建议采用两层防重：

1. 数据库层建立 `unique(order_no, sku_id, ship_time)` 或更细的 `unique(order_no, sku_id, product_no, ship_time)`。
2. 导入流程中执行全历史重复校验，检查 `order_no + sku_id` 是否已经存在。

核心索引建议：

```sql
PRIMARY KEY (id, ship_time)
INDEX idx_ship_time (ship_time)
INDEX idx_product_no_time (product_no, ship_time)
INDEX idx_sku_time (sku_id, ship_time)
INDEX idx_category_time (category, ship_time)
INDEX idx_platform_shop_time (platform, shop_name, ship_time)
INDEX idx_customer_time (customer_no, ship_time)
INDEX idx_logistics_no (logistics_no)
```

导入临时表 tmp_order_import 不建索引，用于提高百万级数据装载速度。数据进入临时表后，再通过 SQL 批量校验和批量写入主表。

## 7. 导入流程

标准月度导入流程：

1. 创建导入批次号，例如 `202406_order_import_001`。
2. 清空 tmp_order_import。
3. 读取 Excel 并写入 tmp_order_import。
4. 校验字段完整性、数据类型、必填项。
5. 校验利润口径：Excel 利润与系统计算利润差异超过阈值时标记异常。
6. 校验本批次重复数据。
7. 校验历史重复数据。
8. 生成异常报告，支持下载。
9. 人工确认后批量写入 t_order_sku_detail。
10. 写入 t_import_log。
11. 导入完成后生成本月汇总快照或缓存统计结果。

## 8. 报表与分析口径

建议内置 8 类报表：

1. 月度经营总览：销售额、销量、成本、费用、利润、利润率。
2. 产品经营排行：按产品、SKU、货品编号统计。
3. 产品月度趋势：按产品和月份对比。
4. 大类经营分析：按大类统计收入和利润贡献。
5. 平台店铺分析：按平台、店铺统计。
6. 区域销售分析：按省、市统计。
7. 物流费用分析：按物流方式、物流费用结构统计。
8. 异常数据分析：亏损订单、低利润产品、高费用订单、重复导入风险。

报表口径需要固定，尤其是利润、费用率、利润率。建议在系统配置中展示口径说明，但不允许普通用户随意修改。

## 9. 技术选型建议

低复杂度落地方案：

| 层级 | 建议 |
| --- | --- |
| 数据库 | MySQL 8.x |
| 后端 | Python FastAPI 或 Java Spring Boot |
| 前端 | Vue 3 或 React |
| 文件处理 | 后端流式读取 Excel，百万级数据建议转换 CSV 后批量导入 |
| 部署 | 单机部署即可，后续再扩展备份机 |
| 权限 | 管理员、导入员、查询员、只读分析员 |

如果团队偏向快速落地，建议 Python FastAPI + MySQL + Vue 3；如果已有 Java 运维体系，建议 Spring Boot + MySQL + Vue 3。

## 10. 性能与长期存储策略

1. 按月分区，所有查询强制带 ship_time 范围。
2. 每月提前创建未来 3 个月分区。
3. 永久保留所有历史分区，不执行 DROP PARTITION。
4. 不做单条循环插入，必须使用批量导入。
5. 不随意增加索引，索引只围绕查询高频维度建立。
6. 每月导入完成后，导出当月 CSV 离线备份。
7. 每周执行整库备份，至少保留最近 4 周。
8. 重要数据建议异地备份。

## 11. 权限与安全

权限建议采用“角色权限 + 数据范围权限”两层控制。角色决定用户能做什么，数据范围决定用户能看哪些部门、平台渠道和店铺的数据。

角色建议：

| 角色 | 权限 |
| --- | --- |
| 管理员 | 用户管理、系统配置、全部数据、全部数据范围 |
| 导入员 | 上传、校验、确认导入、查看导入日志，可按授权范围查看导入结果 |
| 分析员 | 查看看板、报表、导出分析结果，只能查看授权范围内数据 |
| 只读用户 | 查询明细和看板，不允许导入和删除，只能查看授权范围内数据 |

数据范围维度：

| 维度 | 对应订单字段 | 控制说明 |
| --- | --- | --- |
| 部门 | dept | 限制用户只能查看被授权部门的数据 |
| 平台渠道 | platform | 限制用户只能查看被授权平台渠道的数据 |
| 店铺 | shop_name | 限制用户只能查看被授权店铺的数据 |

数据范围规则：

1. 管理员默认拥有全部部门、全部平台渠道、全部店铺数据。
2. 普通查看人员必须配置至少一个可见范围，否则默认不可查看业务数据。
3. 用户可同时绑定多个部门、多个平台渠道、多个店铺。
4. 查询、看板、报表、明细导出都必须自动附加数据范围条件。
5. 如果同时配置部门、平台渠道和店铺，系统按交集过滤，例如只能看“授权部门 + 授权平台 + 授权店铺”的数据。
6. 店铺通常归属于平台渠道，配置时应避免授权了店铺但未授权对应平台渠道导致查不到数据。
7. 导入员导入数据不应绕过数据权限；导入完成后的预览、异常明细和导入结果查看也应按授权范围过滤。

建议增加权限配置表：

| 表名 | 用途 |
| --- | --- |
| t_user | 用户基础信息 |
| t_role | 角色定义 |
| t_user_role | 用户角色关系 |
| t_user_data_scope | 用户数据范围，记录用户可查看的部门、平台渠道、店铺 |

`t_user_data_scope` 可采用如下逻辑字段：

```text
user_id
scope_type: dept / platform / shop
scope_value: 具体部门、平台渠道或店铺名称
```

后端所有查询统一追加权限条件：

```sql
WHERE ship_time >= :start_time
  AND ship_time < :end_time
  AND dept IN (:authorized_depts)
  AND platform IN (:authorized_platforms)
  AND shop_name IN (:authorized_shops)
```

如果某个维度授权为“全部”，则该维度不追加过滤条件；如果某个维度为空且不是全部权限，则直接返回空结果。

安全要求：

1. 导入、导出、删除、配置修改必须记录操作日志。
2. 收货人和电话属于敏感信息，默认在列表中脱敏展示。
3. 导出明细时需要权限控制。
4. 系统不提供物理删除历史数据功能。
5. 所有查询接口必须在服务端执行数据范围过滤，不能只依赖前端隐藏菜单或筛选项。
6. 用户权限变更需要记录操作人、变更前范围、变更后范围和变更时间。

## 12. 实施计划

### 第一阶段：数据底座与导入闭环

周期：1 到 2 周

交付内容：

1. MySQL 表结构与分区方案。
2. Excel 字段映射。
3. 月度导入功能。
4. 数据校验、重复校验、异常报告。
5. 导入日志。
6. 订单明细查询和导出。

验收标准：

1. 可成功导入当前 Excel 模板。
2. 重复数据可识别并阻止入库。
3. 异常数据可定位到具体行和原因。
4. 明细查询必须支持按月查询。

### 第二阶段：产品业务分析

周期：1 到 2 周

交付内容：

1. 首页经营看板。
2. 产品销量、销售额、利润排行。
3. 产品月度趋势。
4. 大类分析。
5. 低利润产品预警。
6. 平台店铺分析。

验收标准：

1. 指标口径和 Excel 人工汇总结果一致。
2. 支持按月份、平台、店铺、产品筛选。
3. TOP 产品和亏损产品可追溯到订单明细。

### 第三阶段：区域、物流、成本分析

周期：1 周

交付内容：

1. 省市区域分析。
2. 物流方式和费用结构分析。
3. 异常费用订单识别。
4. 客户维度分析。

验收标准：

1. 可发现高费用、高亏损、低利润率订单。
2. 可按区域、物流方式、客户追溯明细。

### 第四阶段：稳定性与运维

周期：持续优化

交付内容：

1. 自动创建未来分区。
2. 备份脚本。
3. 导入性能优化。
4. 慢查询优化。
5. 权限和脱敏完善。

验收标准：

1. 百万级月度数据导入流程稳定。
2. 常规月度报表查询可在可接受时间内返回。
3. 备份可恢复验证通过。

## 13. 风险与注意事项

1. Excel 当前只有字段，没有样例数据；正式开发前需要至少提供 1 到 3 个月真实样本，用于验证字段类型、空值、重复和利润口径。
2. `订单编号 + SKU` 是否真的唯一需要用历史数据验证。如果存在同一订单同一 SKU 多行拆分，需要调整唯一键口径。
3. Excel 中“利润”字段不要盲信，建议作为校验值，不作为唯一利润来源。
4. 收货人和电话属于敏感数据，需要脱敏和导出权限。
5. 如果未来要做实时订单同步，再考虑接口表、消息队列或增量同步；当前月度 Excel 导入场景不需要复杂化。

## 14. 最终建议

先按“导入闭环 + 明细查询 + 产品分析”做第一版。只要第一版把字段映射、利润口径、重复校验、月度分区和产品经营指标打牢，后续增加区域、物流、客户、店铺分析都会很顺。

本项目的关键不是把系统做复杂，而是把订单明细长期存稳，把产品业务指标算准，把每一次导入都能追溯。
