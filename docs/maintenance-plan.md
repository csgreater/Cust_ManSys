# 维护更新计划

> 制定日期：2026-07-17。计划基于当前代码仓库与工作区变更，不替代对生产数据库、服务器、备份和访问日志的现场检查。

## 现状与主要风险

系统的订单导入、权限隔离、经营分析和 Vue 运营台已经具备可用基础，线上部署与数据库检查也已有文档和脚本。当前需要从“功能交付”转入“可持续运营”，优先处理以下风险：

| 优先级 | 风险/缺口 | 影响 | 建议处置 |
| --- | --- | --- | --- |
| P0 | Coding Plan 相关文件仍未提交 | 版本不可追溯，发布范围不清晰 | 已完成本地测试与安全收口；下一步是代码审查后形成单一提交，未验收前不要部署到生产。 |
| P0 | Coding Plan 的模型成本需要跨进程/网关级防护 | 应用内额度在多进程部署时无法作为全局硬限制 | 已改为仅登录用户可访问，并增加按用户/IP 限流、每日额度和脱敏调用审计；生产网关仍需配置全局限流、额度与告警。 |
| P0 | 基础自动化测试与 CI | 回归问题会在手工操作或生产暴露 | 已新增单元测试及 GitHub Actions 编译/测试/构建检查；下一步是在仓库启用分支保护并补 API 冒烟测试。 |
| P1 | 迁移脚本按功能散落，缺少发布版本与执行记录 | 漏迁移或顺序不一致会导致线上结构漂移 | 建立迁移清单、执行日志和升级/回滚 runbook；每次发布明确“从哪个版本升级”。 |
| P1 | 备份策略在文档中有建议，但未见恢复演练证据 | 故障时无法确认 RTO/RPO | 固化备份任务、异地副本和季度恢复演练，记录结果。 |
| P2 | 依赖只设置了最低版本范围，缺少自动审计机制 | 安全修复和构建可复现性不足 | 用锁文件和定期依赖审计；Python 依赖在验证后锁定版本范围。 |

## 分阶段执行

### 阶段一：发布基线与安全收口（0–2 周）

目标：任何生产改动都可追溯、可验证、可回退。

1. 已完成 Coding Plan 的登录保护、配置缺失保护、输入长度限制、按用户/IP 的分钟限流、每日额度和安全错误处理；上线前补充真实网关超时与额度验收。
2. 在反向代理或模型网关设置全局额度/告警，避免多进程部署绕过应用内每日计数。
3. 每次发布使用固定清单：代码版本、环境变量变更、前端构建、数据库备份、迁移脚本、健康检查、回滚动作和负责人。
4. 将敏感日志最小化：不记录 API 密钥、会话 Cookie、完整用户地址/手机号和完整模型对话；如需排障，保留脱敏后的请求标识与错误摘要。

验收标准：生产启动配置检查通过；`/healthz` 与 `/readyz` 返回 200；未认证访问不能调用受限的模型接口；发布可在演练环境按 runbook 回退。

### 阶段二：测试与交付流水线（2–4 周）

目标：核心流程在合并前自动发现回归。

1. 已覆盖导入字段映射/校验和 Ark 响应解析的单元测试；继续为利润计算、日期范围、权限范围、脱敏和文件去重补充单元测试。
2. 为登录、订单查询、导入“上传—校验—确认入库”、自然语言分析 SQL 审计、Coding Plan 配置/限流编写 API 冒烟测试；测试数据库必须与生产隔离。
3. CI 已执行 Python 编译、后端测试、自然语言分析自检、`npm ci` 和 `npm run build`；下一步增加依赖漏洞审计并启用合并门禁。
4. 保留一个去敏的标准 Excel 样本，覆盖正常行、警告行、异常行、重复批次和大字段场景。

验收标准：主分支每次提交均有可复现的后端测试和前端构建结果；关键导入及权限场景具备自动化覆盖；失败的流水线禁止发布。

### 阶段三：数据库与可观测性（1–2 个月）

目标：数据可恢复、性能可预警、迁移可审计。

1. 建立版本化迁移台账：发布日期、脚本名、适用起止版本、前置备份、执行人、执行时间、验证 SQL、回滚方式。
2. 建立每日全量备份或“全量 + binlog”策略，并保留异地/对象存储副本；每季度在独立实例恢复一次，记录恢复时长和抽样校验结果。
3. 记录基础指标：接口错误率、`/readyz` 失败、导入耗时/失败率、数据库连接错误、慢查询、磁盘空间、模型调用次数和费用。
4. 对大导入建立容量基线：行数、文件大小、解析时长、入库时长、失败原因；超过基线时优先检查索引、分区和超时配置。

验收标准：任意一次线上变更都能追溯数据库结构；可在目标恢复时间内恢复一次备份；健康、导入和模型异常都有负责人可接收的告警。

### 阶段四：持续优化（持续）

1. 每季度评审依赖升级、安全公告、权限范围和默认账号状态。
2. 每季度检查页面/API 使用情况，清理已废弃兼容页面或将其明确标注为维护模式。
3. 在真实运营数据稳定后，评估报表预聚合、异步任务队列和导入任务持久化，以降低大批量导入对 Web 进程的压力。

## 运行节奏

| 节奏 | 必做事项 | 产出/证据 |
| --- | --- | --- |
| 每日 | 检查备份任务、`/healthz`、`/readyz`、错误日志与磁盘空间 | 监控记录或值班确认 |
| 每次发布 | 备份、迁移清单、构建、自动化测试、健康检查、回滚确认 | 发布记录 |
| 每月导入前 | `ensure_future_partitions.py`、`check_db_ready.py`、确认空间和 Nginx 上传限制 | 导入前检查结果 |
| 每月导入后 | 抽样核对订单数/金额、检查异常批次、按策略清理导入暂存数据 | 导入核对表 |
| 每季度 | 恢复演练、权限复核、依赖审计与升级评估 | 演练报告和整改项 |

## 建议的发布门禁

发布前应同时满足：

- 工作区无与本次发布无关的未提交改动；发布提交和版本号明确。
- `.env` 不入库，生产密钥、Cookie 配置、默认密码和 MySQL 应用账号均通过配置校验。
- 前端构建成功；后端测试、导入回归和权限回归通过。
- 已执行数据库备份；本次迁移与当前线上版本匹配并写入台账。
- 部署后健康检查通过，并完成管理员、导入员、分析员、只读用户四类角色的关键路径抽测。
